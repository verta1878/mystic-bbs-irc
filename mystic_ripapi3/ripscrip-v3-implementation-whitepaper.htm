1. Introduction

This document describes the Mystic BBS IRC Fork implementation of RIPscrip v3.0, a server-side rendering engine for the Remote Imaging Protocol Scripting Language. It is a companion to the original RIPscrip 3.0 Technical White Paper by Jeff Reeder (TeleGrafix Communications, December 1996), which described the planned v3.0 specification.

TeleGrafix dissolved before completing RIPscrip 3.0. Their whitepaper described the vision; this document describes what we built. Where TeleGrafix's plan was a client-side terminal protocol, our implementation is a server-side rendering engine that produces pixel buffers from RIP commands. The server owns all rendering; the client displays pixels and handles user input.

1.1 Scope

The engine processes RIPscrip v1.54, v2.0, and v3.0 commands, renders to an in-memory pixel buffer (Indexed8, RGB24, or RGB32), and exports to BMP. It is designed for integration into BBS software (Mystic BBS) but compiles as a standalone unit with zero external dependencies.

1.2 Lineage

  ripscript.pas (v1.54)  →  rip2api.pas (v2.0)  →  rip3api.pas (v3.0)
      4,041 lines              5,173 lines             8,293 lines

Each engine is backward-compatible with the previous version. A v3.0 engine can render any v1.54 or v2.0 scene file.

1.3 Platforms

All FPC 2.6.4irc targets: Win32, Linux (x86/x64), FreeBSD, Darwin, DOS (DPMI), OS/2. Pure Pascal — no ASM, no C, no external libraries. Compiles with a single command:

ppcx64 -Mdelphi -Fuimg -Fuwav -Fuprg -Fupasjpeg rip3api.pas

2. Architecture

2.1 Server/Client Split

Unlike TeleGrafix's client-side terminal approach, our engine uses a server/client architecture:

  SERVER (BBS host)                    CLIENT (terminal)
  ─────────────────                    ─────────────────
  TRIPEngine instance                  Pixel display
  Processes RIP commands               Receives rendered pixels
  Manages pixel buffer                 Handles keyboard/mouse
  Audio state management               Audio playback (wavplay)
  Table/form data model                Form input (text fields)
  Variable storage + scoping           Variable display
  Progressive rendering                Progressive display

The server owns all data and rendering state. The client is a thin display layer. This simplifies BBS integration — the server generates the scene, the client shows it.

2.2 Pixel Buffer

The engine supports three pixel formats, switchable at runtime:

FormatBytes/pixelColorsBuffer
Indexed81256 (palette)Pixels^[Y,X]
RGB24316.7 millionPixelsRGB
RGB32416.7M + alphaPixelsRGB32

All drawing primitives work across all formats. In Indexed mode, the engine uses palette lookup for backward compatibility with v1.54 scenes. In RGB modes, colors are specified directly.

2.3 Codec Architecture

117 codec files organized in four directories:

  img/  — 20 image, animation, and graphics codecs
  wav/  — 33 audio, streaming, and utility codecs
  prg/  —  6 progressive rendering codecs
  pasjpeg/ — 58 JPEG library files (IJG port)

All codecs are standalone Pascal units. They operate on raw byte buffers and pixel arrays — no framework coupling, no TStream, no Classes unit. Each codec compiles independently.

3.6 Streaming Audio (Whitepaper §4.3)

IMPLEMENTED — exceeds specification

TeleGrafix planned "compressed, streaming audio" as a future goal. We implement a full audio subsystem:

  4-stream simultaneous playback
  13 audio format decoders (WAV, VOC, MP3, FLAC, AIFF, AU,
    MOD, S3M, XM, MIDI, ANSI music, ADPCM + LZMA compression)
  WAV streaming: WAVStreamStart → Feed → End (chunked)
  Cue points: CueAdd(frame, action) — 64 timed events
  Background audio with crossfade transitions
  Ring buffer for streaming playback (ringbuf.pas)
  Network audio streaming (netaudio.pas)

3.8 Full-Motion Video (Whitepaper §4.2)

DEFERRED TO v4.0

TeleGrafix planned full-motion video as a future goal. Our v3.0 engine provides animation support through frame counters, cue points, and progressive rendering codecs. Full video codec support (AVI/MPEG decoding) is deferred to v4.0.

3.9 Document Oriented Interface (Whitepaper §4.6)

DEFERRED TO v4.0

TeleGrafix envisioned a page layout engine within RIPscrip. Our v3.0 table and form system (Phase 20) provides structured data presentation. A general-purpose document layout engine is deferred to v4.0.

3.10 HTML Rendering (Whitepaper §4.7)

DEFERRED TO v4.0

TeleGrafix planned to embed HTML rendering within RIPscrip 3.0. Modern BBS clients (SyncTERM, NetRunner) handle HTML separately. HTML rendering within the engine is deferred to v4.0.

3.11 MIDI Synthesis

IMPLEMENTED

The engine parses MIDI files via mididec.pas and synthesizes audio via midisynth.pas — a 2-operator FM synthesis engine (OPL2/AdLib style) with 16 MIDI channels. midisynth.pas is {$H-} compatible and renders MIDI events directly to PCM. midiplay.pas bridges the parser and synth for file playback (uses {$H+}, host-side). The engine stores the filename via MIDILoad; the host can use midiplay.pas for full playback or midisynth.pas directly for real-time note events.

3.12 Full Unicode

DEFERRED

The engine maps UTF-8 to CP437 via UTF8ToCP437 (full 256-char mapping). A rip3unicode.pas companion unit is tracked for future implementation, providing full Unicode pass-through for RFF fonts with wider character ranges. Current CP437 mapping is sufficient for BBS use.

4. Progressive Rendering

This is entirely new — not in the TeleGrafix whitepaper. Over slow modem connections, the user sees partial results as data arrives, rather than waiting for the complete scene.

Five progressive rendering codecs in the prg/ directory:

CodecStrategyBest For
ripdecraw.pasIncremental stream (cmd-by-cmd)Interactive, fast connections
ripbindec.pasBinary scene (compact pre-compiled)Cached menus, splash pages
riptile.pasTile-based (64×64 mosaic)Full-screen art, slow modems
riplayerdec.pasLayer-based (bg → overlays)Complex UI with background
ripchange.pasChanged-region patchesAnimations, live updates

Additionally, riprender.pas provides a unified progressive renderer that integrates all five codecs with Phase 23 graphics.

Image streaming codecs complement progressive rendering:

  • jpegprog.pas — progressive JPEG (scan-by-scan)
  • gifinterlace.pas — interlaced GIF (4-pass)
  • pnginterlace.pas — Adam7 interlaced PNG (7-pass)
  • gifanim.pas — animated GIF frame playback
  • spriteanim.pas — sprite sheet animation

8. What TeleGrafix Got Right

Reading the 1996 whitepaper thirty years later, TeleGrafix's vision was remarkably prescient:

Their emphasis on text-based commands over binary protocols (§3.2) proved essential for BBS compatibility — our ProcessLine function still parses the same pipe-delimited text format.

Their image format support (§3.7) correctly predicted that JPEG, GIF, and PNG would dominate. We added eight more formats, but those three remain the most-used.

Their data tables (§3.8) anticipated that BBS software would need structured data presentation beyond simple text and graphics — file listings, user lists, message headers.

Their streaming audio (§4.3) was the right call for modem connections. Our ring buffer and chunked streaming design follows exactly the architecture they described.

Where they overreached was HTML rendering (§4.7) and full-motion video (§4.2) — both reasonable goals for 1996, but better handled by dedicated software (web browsers, media players) than by a graphics terminal protocol.

10. Known Limitations

• CompositAlpha requires a full canvas-sized source buffer (FXAlphaBlend operates on the complete pixel array).

• mididec.pas uses Classes/TStream — incompatible with the engine's {$H-} mode. The engine stores the MIDI filename; the host application handles actual MIDI parsing and playback.

• RFF stroke rendering uses lines only (MoveToEx/LineTo) — no curves. This is correct per the RIPaint 2.0 GDI analysis (RIPAINT_FINDINGS.md).

• Full Unicode is deferred. Current CP437 mapping covers all 256 characters needed for BBS use.

• TRIPVariable.Name is String[31] (was String[12] in v1.54, expanded to support form field names like LOGIN_REMEMBER).
