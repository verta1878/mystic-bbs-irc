================================================================
 Mystic BBS 1.10 A38 - IRC whatsnew
================================================================
 This file lists changes made in this fork of the 1.10 A38 source.
 Ordered OLDEST FIRST: the initial-port Phase 0 work is at the top;
 the most recent work is at the bottom.  Convention (matching Mystic's
 own whatsnew):
    +  new feature / addition
    !  bug fix / behavior change
 Upstream socket-layer history is in CHANGELOG.TXT (V. Chapman, 2013).

 The 'IRC' name is short for Internet Relay Chat - a nod to the era that
 carried this scene forward (modems -> BBS -> IRC -> Discord).  It also
 marks why this fork exists at all: the open GPL release of the A38 source
 is what allowed the code to live.  Release it, and it survives.

 Phase 0 - initial port to buildable state (FPC 2.6.2, win32 + i386-linux)
 These are the foundational fixes that got pristine A38 building as native
 32-bit binaries on both target platforms (13/13 each).

 ! source encoding: 9 .pas files had been corrupted by a Latin-1 -> UTF-8
   round-trip, turning single-byte CP437 box/block chars (e.g. $C4 '-') into
   2-byte UTF-8 sequences that broke both compilation (char literals) and
   CP437 rendering.  Restored byte-for-byte via utf8->latin-1 (verified by
   round-trip): bbs_ansi_menuform, mide, 109to110, bbs_cfg_msgbase,
   bbs_cfg_menuedit, mystpack, bbs_filebase, bbs_cfg_echomail, bbs_cfg_theme.
   The tree's original #nnn CP437 convention is preserved; no codepage flag.

 ! m_ops.pas: fixed a malformed conditional {$IFDEF CPUX86_64 (missing closing
   brace).  Dormant on x86_64 but an ACTIVE compile error on i386 - critical
   for the 32-bit targets.

 ! exec: Unix.Shell was removed from FPC 3.x (still present in 2.6.x).  Made
   the three call sites version-conditional so the tree builds on BOTH:
   {$IF FPC_FULLVERSION >= 30000} fpSystem(...) {$ELSE} Shell(...) {$IFEND}
   (bbs_database, bbs_common, mide).

 ! win32 build blocker: m_io_sockets / m_socket_class had UNCONDITIONAL
   {$link m_resolve_address.o} + {$linklib c} + an external Unix getaddrinfo
   resolver, which breaks any Windows link.  Guarded the Unix C-glue with
   {$IFDEF UNIX} (Unix path unchanged) and added a {$IFDEF WINDOWS} XP-safe
   IPv4 resolver via Winsock2 gethostbyname (dotted-quad built manually to
   avoid the Windows/Winsock2 duplicate-type clash), matching the C glue's
   contract.  (IPv6-on-win32 via WSAAddressToString is a later item; inet_ntop
   is Vista+.)

 ! 109to110: the migration tool wrote the renamed config field inetIPLogging;
   updated the write to inetLogging (the current records.pas name), keeping the
   old 1.09 field name only on the read side.

 + build: the IPv6 resolver C glue (m_resolve_address.c, V. Chapman 2013) is
   compiled to m_resolve_address.o as part of the Unix build (cc -c -fPIC ...
   + -Fomdl).  Added build.sh (Linux) and build-win32.bat (native XP).  Stale
   .ppu/.o are cleaned between runs to avoid spurious 'cannot open file'
   errors.


================================================================
 Phase 1 - RIPscrip graphics (mystic_rip/ example module)
================================================================

 + rip: RIPscrip graphics support, phase 1.  The fork now has a working RIP
   terminal engine, as a self-contained example module (mystic_rip/) beside
   the other add-ons.  TTermRip (mystic_rip/rip_term.pas) is the exact
   parallel of the ANSI class TTermAnsi: feed it a byte stream and it
   interprets RIPscrip 1.x drawing commands (lines, boxes, bars, circles,
   ovals, flood fill, text, pixels, and mouse hot-region buttons).  Because
   RIP is vector graphics, TTermRip renders to a graphics seam (rip_canvas)
   instead of the text screen; a pure-software EGA surface (rip_surface)
   implements it, and an SDL2 window presenter (rip_window, runtime-loaded
   SDL via mystic_sdl's binding) shows it with working clickable buttons.
   Two demos ship: rip_render turns a .RIP file into a BMP with no display
   at all, and rip_view shows it in a window where clicking a RIP button
   prints what it would send to the host.  The engine is the maintainer's
   clean-room ripterm client lifted into the tree - built once, used twice -
   and the in-tree renderer's output is pixel-identical to the standalone
   client's.  The module uses FPC RTL units only (no mdl units); when RIP
   goes live in the core, rip_term.pas is promoted to mdl/m_term_rip.pas
   beside m_term_ansi.pas (docs/RIP-INTEGRATION.md maps it).  Fill patterns,
   fonts, the full button/icon system, and serving RIP screens are the next
   phases.

================================================================
 DESIGN NOTES (why certain choices were made)
================================================================

 * Startup path-recovery prompt input method:
   The path-recovery prompt (CheckDIR) uses a hand-rolled line reader
   built on the low-level console/keyboard primitives (Keyboard.ReadKey
   + Console.WriteChar) rather than Mystic's full input field
   (Session.io.GetInput / ANSI menu input).

   Reason: CheckDIR runs deliberately EARLY in startup - before the
   crash-prone init (users.dat rewrite / DirCreate) - so a missing or
   moved path is caught before it can crash the program.  At that point
   Session.io, theme data and the ANSI menu system are not yet
   initialized, and the full input field depends on all three.  (A plain
   Pascal ReadLn also fails there, because the console layer is already
   active and intercepts stdin - that was the 'prompt accepts no input'
   bug.)  The low-level primitives are up at that stage, so we use them
   and accept a basic editor (type / backspace / enter).

   The richer alternative would be to BOOTSTRAP the I/O stack (Session.io
   + theme + ANSI) early, purely so this rare recovery prompt could use
   the full field editor.  That is interesting but disproportionate and
   would add early-startup fragility.  Deliberately NOT done, and NOT
   planned for current or future work.  Documented here so the choice is
   known rather than looking accidental.

 * Deferred path-validation enhancements (noted in CheckDIR/CheckPaths):
   - Validate DATA STRUCTURES within each path (expected .dat files exist
     and are the right record size), not just that the directory exists.
   - Proper FidoNet path (Inbound/Outbound) validation - deferred to the
     FidoNet work.  Currently these get only the generic directory check
     (and blank ones are skipped).

 + qwkpoll: now logs to qwkpoll.log in the logs directory (timestamped,

   import/export results/failures are all recorded, so a poll can be
   diagnosed after the fact instead of the output being lost.  Mirrors the
   fidopoll.log pattern.

 ! QWK: messages larger than 64KB were corrupted in packets.  The message
   block-count accumulator was a 16-bit Word (a 16-bit DOS-era assumption)
   and overflowed past 65535 bytes, writing a wrong 128-byte block count to
   the packet header - which desynchronised the reader and cascaded into the
   rest of the packet.  Widened to LongInt (verified against the QWK layout
   spec: correct = ceil(bytes/128)+1 at all sizes).  The QWK import side had
   the mirror bug (a SmallInt block counter that wrapped negative past ~4MB,
   skipping the message); widened its counter and loop variable to LongInt
   too, so write and read agree at every size.

   silently.  It is now guarded and logs the reason on failure.

================================================================
 File descriptions, platform libraries, and more build targets
================================================================

 + file listings: colour in FILE_ID.DIZ descriptions is now preserved, matching
   Mystic 1.12.  Older A38 stripped every control character out of a DIZ as it
   was read, which flattened any colour an author had put in it.  The reader now
   keeps Mystic pipe colour codes as-is and converts embedded ANSI colour into
   pipe codes, so a colourful FILE_ID.DIZ shows up in colour in your file
   listings.  Applies to both normal uploads and the mass-upload utility.

 + file listings: added the Mystic 1.12 full-screen FILE_ID.ANS archive viewer.
   When a user opens an archive in the file base, the BBS now shows the
   FILE_ID.ANS "cover screen" that authors embed alongside FILE_ID.DIZ - a full
   ANSI advertisement for the file, up to 99 lines - before listing the
   contents.  Press A in the archive viewer to show it again.  It reuses the
   same extraction path the DIZ reader uses, so any archive type your BBS can
   already open is supported.

 + platform libraries: the optional add-on modules (SDL2, Hunspell, cryptlib)
   now ship with prebuilt libraries for all three graphical targets, under
   libs/ - Windows (SDL2.dll, libhunspell32.dll, cl32.dll), Linux i386, and
   macOS i386.  Each is loaded at run time and stays off if absent, exactly as
   in stock Mystic.  The Windows cryptlib (cl32.dll) is cross-built and, under
   test, reports "Cryptlib detected and initialised - secure sessions
   available".  The macOS libraries are period-correct builds (SDL2 2.0.1,
   Hunspell 1.6.2) so they build against the 10.6 SDK that supports the widest
   range of Macs.

 ! add-on modules on OS/2: the SDL2 / Hunspell / cryptlib bindings now compile
   for OS/2.  They previously assumed only Unix or Windows for loading a library
   at run time and failed to build on OS/2; they now use the native OS/2 calls
   (DosLoadModule / DosQueryProcAddr / DosFreeModule) as well.

 + build targets: OS/2 and DOS join Windows, Linux and macOS.  OS/2 (i386-os2)
   compiles from the same source with the same FPC 2.6.2; the final link is a
   native OS/2 step, as it always has been for Free Pascal.  DOS is the classic
   32-bit protected-mode target (go32v2, DPMI extender) - built with the same
   2.6.2 as every other target so its data files stay compatible.  A DOS program
   builds to a real "MZ for MS-DOS, DJGPP go32 DOS extender" executable; a full
   DOS port of the BBS itself is a later step, since DOS handles connections
   differently (FOSSIL / door style) than the other platforms.

 + release packaging: each platform's release is built separately and carries
   its own installer content (install_data.mys), a FILE_ID.DIZ naming the target
   (win / lnx / os2 / mac / dos BINARIES), the WHATSNEW and UPGRADE text, and
   the GPL licence - never one combined archive.

 + OS/2: the final link now runs ENTIRELY ON LINUX. A self-hosted emx
   cross-toolchain (patched GNU binutils with a new "a.out-emx" target + the
   emx IMPORT# symbol patch, a Linux port of emxbind, the emxl.exe loader, and
   a data-alignment ld wrapper) turns FPC's a.out into a native OS/2 LX .exe
   on a Linux host - no OS/2 machine needed. All 14 programs build and verify
   as "LX for OS/2 ... emx 0.9d". Build the toolchain from
   libs/os2-linux-toolchain.zip; full docs in docs/os2-linux-toolchain/. This
   is the step ArcaOS/bitwiseworks track as an open problem.
 ! OS/2 is now a FULL 14/14 release (incl. the networked programs and a
   runnable install) built on Linux via LINK=1 ./build-os2.sh.
 + DOS/go32v2: added the missing DOS platform branches (m_ops, records,
   m_fileio, m_output, m_input) - the 7 non-networked utilities now build from
   libs/dos-toolchain.zip. The networked programs await a DOS socket layer
   (Watt-32 or FOSSIL); DOS port otherwise on hold.

 + DOS/go32v2 networking: the DOS build now includes a complete TCP/IP socket
   layer, so the BBS server itself and the networked tools compile for DOS.
   The socket code is written in Pascal and speaks the same Sockets API the
   other platforms use, sitting on the Watt-32 stack + a DOS packet driver at
   run time (the way DOS networking has always worked).  DOS now builds 10 of
   14 programs, INCLUDING the mystic server; the four remaining tools
   (mis / fidopoll / nodespy / qwkpoll) compile and are ready to link once
   Watt-32 (libwatt.a) is supplied.

 ! DOS/go32v2 linking: fixed the toolchain so GNU ld can read Free Pascal's
   DOS object files.  FPC marks its program sections with a symbol class that
   stock binutils rejected ("unrecognized storage class 104"), which blocked
   linking any DOS program that calls a C library.  A small binutils patch
   teaches the DOS linker to accept it - shipped in libs/dos-binutils-patch/
   (patch + the full binutils 2.30 source), and already baked into the bundled
   libs/dos-toolchain.zip so the repo builds DOS from its own tools.

 + DOS build script: build-dos.sh cross-builds every DOS program from the
   bundled toolchain on a Linux host (./build-dos.sh, or one target by name).
   Set WATT32LIB=<dir with libwatt.a> to also link the networked tools.

 + DOS telnet server: on DOS the telnet listener serves one caller at a time,
   handing the connection straight to a mystic node - the single-tasking model
   DOS uses (no threads).  Watt-32 itself supports several connections at once;
   a cooperative multi-node poll loop is planned as a later step.

 + build every release at once: make_all_releases.sh now covers all five
   targets (Linux, Windows, macOS, OS/2 and DOS).  Each target is independent -
   if a toolchain is missing that one is skipped and the rest still build - and
   every target produces the usual FULL and UPGRADE archives.

 + Echomail nodes can now have a maximum PKT size and a maximum ARCmail/bundle
   size, both configured per node in KB (0 = no limit).  When exporting, Mystic
   starts a new PKT once the current one reaches the PKT limit, and rolls to a
   new bundle once the current bundle reaches the ARCmail limit.  The two new
   fields appear in the Echomail Node editor as "Max PKT KB" and "Max ARC KB".

 + NetMail message bases no longer need to be linked/tagged to an echomail node
   to be exported.  Netmail now routes purely by destination address, so a
   NetMail base with no echo tag will still export correctly.

 + When routing NetMail, Mystic now first compares the destination address
   against all configured echomail nodes.  If there is a direct match it routes
   the NetMail straight to that node without consulting Route Info; only if
   there is no direct match does it fall back to matching each node's Route Info
   in order.
